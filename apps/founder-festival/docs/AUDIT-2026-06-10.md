# Founder Festival — Architecture, Security, API & Performance Audit

*Prepared for the founder/CEO • Codebase snapshot: `main`, ~71.6k LOC, 466 source files, 114 API routes*

---

## Executive Summary

**Overall health: B+ / strong for a codebase that grew 65%.** The security fundamentals are genuinely good — all 114 API routes carry an in-route auth gate, every public read goes through an explicit allow-list transform, and the money path (Stripe/credits/scoring-tick) is hardened with atomic CAS + idempotent ledger inserts. **All five prior P0 fixes held** across ~150 new commits. There are **no confirmed critical or anonymous-exploitable vulnerabilities.**

The real debt has moved to three places, in priority order:

1. **A spoofable inbound-email surface.** The Resend inbound webhook appends replies to claim threads matched only by a *sequential, guessable* `Request #` with **zero sender verification** — anyone who can deliver mail to the public inbound MX can inject messages into any admin-reviewed claim thread. (Confirmed; severity adjusted high→**medium** — admin console escapes the body and never shows the forged sender, so it's social-engineering, not direct compromise.)

2. **Hot public read paths run uncached full-table scans that scale linearly with the evaluations table.** Every `/profile` view does 3 full-table percentile scans + a 6-deep sequential await waterfall; every leaderboard load fires 4 uncached aggregate scans (incl. a per-row correlated `EXISTS` into `users`); and `(signal_quality, source)` — the predicate behind all of them — **has no index.** This is the thing that will bite as you scale traffic and the table grows.

3. **The public dev API leaks two things it shouldn't.** Exact internal per-row scoring **point values** (your core scoring IP) ship verbatim via `/api/v1/score`, and the free `/api/v1/search` has **no token-count cap** (query-complexity DoS on an unauthenticated-mintable free key).

**Notable refutation:** the loudest "high" finding — *"superadmin-hidden profiles fully exposed via `/api/v1/score`"* — was **refuted** by adversarial verification and dropped. `hide` was only ever a leaderboard-suppression toggle (by explicit design, see `leaderboard.ts:207-209`), and the public profile page already serves the identical scrubbed data to anyone with the URL, no API key needed. The API is at parity with the existing web surface; it does not defeat a privacy control because no such control exists. *(A related real issue — hidden profiles surfacing as named matrix peers — survives at medium and is listed below.)*

**One-line verdict:** Ship-fast hygiene is intact and the prior hardening held — now harden the new email/webhook surface, index the stats predicate, and lock down the public-API IP leak; then pay down the `eval-pipeline.ts` god file.

## Security

> Confirmed issues only, by **adjusted** severity. Prior P0 regression check at the end.

### Medium

**1. Inbound claim-reply webhook — thread injection / cross-thread mis-routing** *(confirmed; adjusted high→medium)*
`src/lib/claim-thread.ts:94-110`, `src/app/api/inbound/resend/route.ts:74-80`

- **Mechanism:** `recordInboundReply` matches a thread solely on `eq(claimThreads.requestNumber, opts.requestNumber)` parsed from the subject (`parseRequestNumber` accepts `(Request #NNNNN)`, 3+ digits), then inserts an inbound message with the caller-supplied `fromEmail`. No check that the sender belongs to the thread's evaluation. `request_number` is a Postgres serial (`schema.ts:1404`) — sequential and enumerable. The Svix signature is sound but only proves the POST came from Resend's inbound pipeline, not who authored the email; Resend Inbound is a public MX.
- **Exploit:** Attacker emails `attacker@evil.com → reply@reply.festival.so` with subject `RE: … (Request #10042)`. Resend POSTs it with a valid Svix signature; the route finds the unrelated victim's thread (#10042) and inserts the attacker's body as an `← reply`. The admin reads it in the Claim Review Console, potentially influencing an approval.
- **Why only medium:** the console renders the body as an auto-escaped React text child (`PendingItemRow.tsx:290`, no XSS) and **never displays the `fromEmail`**, so there's no forged-sender label — impersonation is implicit, not explicit. Impact is bounded to polluting an admin-only surface; approvals are still manual.
- **Remediation:** resolve the thread's verified recipient (`profile_emails` where status=verified / outbound `toEmail`) and reject/flag `fromEmail` mismatches; treat unknown senders as quarantined. Add a unique constraint on the provider message id with `onConflictDoNothing` to also close the replay/idempotency gap (`svix-id` is in scope at `route.ts:51`).

**2. Org-badge create/delete bypasses per-org assignment scope** *(medium)*
`src/app/api/admin/org-badges/route.ts:36-67`, `src/lib/org-badges.ts:46-50`

- The bulk-apply path correctly calls `canApplyOrgBadge(badgeId)` against `adminOrgAssignments`, but POST/DELETE gate **only** on `requireGrant('manage_events')` (a delegatable, non-superadmin grant). A delegated admin assigned to only their own host/sponsor can create badges on, and DELETE badges of, *any* org — and `deleteOrgBadge` cascades `db.delete(badgeOverrides).where(badgeId='org:<id>')` (`org-badges.ts:48`), stripping that badge from **every** profile it was applied to. Admin-to-admin cross-tenant IDOR.
- **Fix:** verify actor assignment to `(ownerType, ownerId)` before `createOrgBadge`; load the badge and check `canApplyOrgBadge(id)` before `deleteOrgBadge` (superadmin bypass).

**3. Event recap/config sub-editors skip `canAccessEvent()` scope check** *(medium — RBAC scope-bypass IDOR)*
12 routes under `src/app/api/admin/events/[id]/*` (`date`, `slug`, `learnings`, `hosts`, `sponsors`, `priorities`, `photos/*`)

- These gate only on `requireGrant("manage_events")` then immediately `db.update(events)…where(eq(events.id, id))`, omitting the `canAccessEvent(id)` ownership check their sibling routes (`attendees/*`, `applicants/*`) apply. An admin with `manage_events` + Events scope = `"theirs"` can POST to any event id and rewrite its slug (breaking shared links), date, recap HTML, hosts/sponsors, or delete/replace photos.
- **Fix:** after `requireGrant`, add `if (!(await canAccessEvent(id))) return 403;` in all 12. Better: a shared `requireEventAccess(id, "manage_events")` helper so grant+scope can never again be applied half-way (mirroring `canAccessJob`).

**4. Hidden profiles surface as named matrix peers via `/api/v1/score`** *(medium)*
`src/lib/founder-matrix.ts:96-107`, `src/lib/api/score-payload.ts:122-139`

- `getMatrixCandidates` filters `ne(signalQuality,'low')` + `ne(source,'code')` but omits `isNull(hiddenAt)` (unlike the leaderboard's `baseWhereFor`). So `toMatch` emits a hidden profile's `full_name`/`profile_href`/`avatar_url` as a similar/opposite peer, leaking the identity of suppressed profiles.
- **Fix:** add `isNull(evaluations.hiddenAt)` to the `getMatrixCandidates` WHERE.

**5. Event chat title/body unbounded + emailed on @mention** *(medium)*
`src/lib/event-chat.ts:241`, `src/app/api/events/[slug]/chat/route.ts:22`, `schema.ts:1333`

- Columns are unbounded `text`; routes only `.trim()` + emptiness-check. An attendee can store multi-MB posts (DB bloat, slow render) and have them re-emailed to mentioned attendees (amplification). Attendee-gated, not anonymous.
- **Fix:** cap title≤200 / body≤5000 server-side, 400 on overflow; slice as backstop in `createComment`.

**6. `score-items modify` accepts unbounded/non-finite points** *(medium — correctness with security impact)* — see P1-5 above.

### Low (hardening / defense-in-depth)

## API (public dev API + the 114-route surface)

**Overall the API layer is unusually disciplined** — and this is a genuine strength worth preserving. Every mutating route uses an explicit field **allow-list** (no mass-assignment), every v1 endpoint hard-requires a Bearer key via `verifyApiKey` (SHA-256 hashed, revoked-aware), paid endpoints (`score` POST, `resolve`) sit behind a global daily circuit-breaker that trips *before* any Exa/Claude spend with idempotent credit reserve/refund, and IDOR-prone account/family/developer-key routes scope DB queries by the viewer's own id.

**The gaps that matter:**

1. **Scoring-IP leak via `/api/v1/score`** — exact per-row `points` exposed (P0-3 above). The headline API exposure issue.
2. **No token cap on free `/api/v1/search`** — query-complexity DoS (P1-4 above). The only paid endpoints have a global breaker; **search has none**.
3. **Hidden profiles as named matrix peers** — Security #4 above.
4. **Unbounded numeric filters → 500 instead of 400** (`src/lib/leaderboard.ts:105-110,172-176`) — `team_min=9999999999` overflows int4 in `(…->>'employeesCount')::int >= ?`, raising an unhandled Postgres error. Clamp `raised_min`/`raised_max`/`team_min` to non-negative bounds like `limit` already is. **(low, S)**
5. **Inconsistent error shape + no versioning envelope** (`v1/score:25,30`, `v1/resolve:33`, `v1/search:32`) — bodies mix snake_case machine codes (`invalid_api_key`) with free prose (`"name (>= 2 chars) required"`) in the same `error` field; success payloads un-enveloped. Consumers can't switch on a stable code. Standardize on `{error:{code,message}}` + an `api_version` field — **lock this in before the surface grows.** **(low, M)**

**Whole-surface consistency (114 routes):**
- **Body-parse inconsistency** — most routes wrap `await req.json()` in try/catch→400, but a few (`connections/respond:10`, `events/[slug]/chat:21`, `…/reply:35`) don't, surfacing malformed bodies as 500s. Adopt a shared `readJson<T>()` helper. **(low, S)**
- **No shared validation/response convention** — validation is hand-rolled per route, which is exactly where the `points` and chat-length gaps slipped in. The existing allow-list discipline makes a thin Zod-per-body + `json()/err()` helper layer a **mechanical** refactor, not a redesign. Roll out starting with `/api/v1` for a stable public contract. **(low, L)**
- **`/api/redeem` mints a synthetic `evaluations` row per accepted code, unauthenticated** (`redeem/route.ts:60`) — rate-limited (15/IP/day, 300 global) and bounded by `max_uses`, but a high-`max_uses` code accretes junk rows. Dedupe per `(code, day)` or document the `max_uses` invariant. **(low, M)**

**Rate-limiting posture is solid** (`rate-limit.ts` per-IP daily + global circuit-breaker; `request-ip.ts` trusts only `x-vercel-forwarded-for`) — the one hole is search having no breaker bucket.

## Performance

The pipeline architecture is right (parallel enrichers w/ per-source timeouts, keyset-paginated leaderboard w/ externally-managed partial indexes, SSR first pages). The problem is **uncached full-table scans on the hottest public reads** plus **client-side rendering of unbounded admin datasets**. Concrete fixes:

### Database / indexing at the new scale

| Issue | Fix |
|-------|-----|
| **No `(signal_quality, source)` index** behind 3 percentile scans/profile + population + matrix (P0-2) | `CREATE INDEX CONCURRENTLY evaluations_scored_pop_idx ON evaluations (founder_score, investor_score, score) WHERE signal_quality <> 'low' AND source <> 'code'`; collapse 3 `computePercentile` calls into one conditional-aggregate pass; memoize denominators like `getPopulation` does. |
| **Full `profile` JSONB blob read per leaderboard row** (P1-2) | Project needed keys via `jsonb_build_object(...)`, or promote `companyName`/`primaryCompanyDomain` to columns (as `canonicalIndustries` already is). |
| **4 uncached leaderboard aggregates/request** incl. correlated `EXISTS` into `users` + JSONB `unnest` (P1-1) | Cache global badge/industry/count aggregates (`use cache` + cacheLife, cacheTag on score writes); materialize the `claimed` count from a join. |
| **JSONB facet filters (raised/team/outcome/badge) + `canonical_industries` overlap unindexed** (`leaderboard.ts:152,164-176`) | Add `CREATE INDEX CONCURRENTLY evaluations_canonical_industries_gin ON evaluations USING gin (canonical_industries)` (cheap, high-value — makes `arrayOverlaps` index-assisted). Expression indexes on numeric facets only if those filters prove common. Revisit the `schema.ts:185` "unindexed for now" TODO now that filtering shipped. |
| **N+1 event-attendee name resolution**, each an unindexed `lower(full_name)` seq-scan (`events.ts:154-162`) | Batch into one `WHERE lower(full_name) = ANY(<names>)`; add `evaluations_full_name_lower_idx ON evaluations (lower(full_name))` (the trigram GIN does **not** serve exact equality). |
| **Search ORs JSONB ILIKE that defeat the trigram index** (`leaderboard.ts:573-585`) | Only `full_name` has a trigram index; the JSONB-extraction ILIKEs force a seq-scan. Add expression trigram GINs on company fields if company search matters, else restrict to indexed columns. |
| **All DB traffic uses non-pooled neon-http single-shot driver** (`db/index.ts:1-6`) | Correct for request routes, suboptimal for crons/bulk (no pipelining, no transactions). Give crons/bulk a pooled/WebSocket connection (`drizzle-orm/neon-serverless` Pool over `DATABASE_URL_UNPOOLED`). At minimum, batch statements to cut round-trips. |
| **`rate_limit` pruned only weekly** (bolted onto `refresh-mm`'s `0 3 * * 0`) | Move the `DELETE` to a daily cron so it tracks its intended 2-day retention. |

## Reliability (newer email surfaces — the weak spot)

The money-path crons are hardened; the **new email surfaces are not idempotent and swallow errors silently**:

- **Inbound webhook not idempotent** — Resend at-least-once retry or a captured-payload replay within the 300s Svix window inserts duplicate thread messages (`claim-thread.ts:87`). Fix with the unique-message-id constraint noted in Security #1.
- **Chat-mention emails: dedup marker committed *before* send + empty catches** (`event-chat-email.ts:53,84,88`) — a transient Resend failure means the email is **never sent and never retried**, and the loss is invisible (no `reportServerError`). Insert the marker *after* successful send (mirror `welcome-email-sweep.ts`); add `reportServerError` to both catches. **(M)**
- **`changelog-notify` marks `notifiedAt` only after the full recipient loop, no concurrency guard** (`changelog.ts:81-108`) — a mid-entry timeout re-emails the *whole* subscriber list next tick. Make the claim atomic: `UPDATE … SET notified_at=NOW() WHERE id=$id AND notified_at IS NULL RETURNING`, send only if the row was claimed. Scales with subscriber count. **(M)**
- **External calls (Twilio/Resend/AnyMailFinder/Stripe) have no per-call timeout** (`sms.ts:46`, `anymailfinder.ts:23`) — a hung upstream wedges a cron tick to `maxDuration`, abandoning the rest of the batch. Add `AbortController` 10-15s timeouts, matching the proven per-enricher pattern. **(M)**
- **Zero test coverage on inbound webhook, `verifySvix`, and `changelog` notifier** — the signature verifier is the *sole* auth gate on a public POST endpoint and is untested. Add unit tests (valid/rotated/expired/bad-base64 sig; idempotent replay; only-unnotified-entries-fire). **(M)**
